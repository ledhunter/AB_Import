# 📚 Документирование изменений проекта

## Триггер
Когда пользователь просит задокументировать изменения или дополнить документацию.

## Описание
Дополни документацию в папке `doc_project` с учетом последних изменений. Документируй используемую реализацию с пояснениями, чтобы в следующий раз не допускать ошибки, которые ломают предыдущую реализацию.

## Шаги

1. **Проанализируй изменения** — определи, какие паттерны, компоненты или решения были реализованы
2. **Проверь существующую документацию** — просмотри файлы в `doc_project/` для понимания формата
3. **Создай или дополни документацию**:
   - Используй emoji для улучшения восприятия (📋, ✅, ❌, ⚠️, 🎯, 🏗️)
   - Структурируй с заголовками и таблицами
   - Добавь примеры кода с комментариями
   - Опиши **правильную** и **неправильную** реализацию
   - Укажи файлы и классы, где применяется паттерн

## Формат документации

```markdown
# 🎨 [Название паттерна/компонента]

## 📋 Описание
Краткое описание назначения и контекста использования.

---

## ✅ Правильная реализация

\`\`\`css
/* Код с комментариями */
.element {
  flex-shrink: 0;  /* 👈 Пояснение */
}
\`\`\`

### ⚠️ Важно
- Ключевые моменты, которые нельзя упустить

---

## ❌ Типичная ошибка

\`\`\`css
/* НЕПРАВИЛЬНО - объяснение почему */
.element {
  /* Нет flex-shrink: 0 — элемент может сжаться! */
}
\`\`\`

---

## 📍 Применение в проекте

| Компонент | Файл | Классы/Переменные |
|-----------|------|-------------------|
| ViewDrawer | gantt.css | .ganttDrawer |

---

## 🎯 Чек-лист
- [ ] Пункт проверки 1
- [ ] Пункт проверки 2
```

## Примеры тем для документации

- CSS паттерны (flex-layout, tabs, scrollbar)
- Компоненты UI (Toast, Modal, Drawer)
- Хуки и состояния
- API взаимодействие
- Типы данных
